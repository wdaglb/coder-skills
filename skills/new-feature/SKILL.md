# New Feature

## 概述

在开始任何实质执行前，先过“需求澄清门”。
先输出一句任务理解和一组执行清单，再判断任务歧义等级，
并识别哪些节点需要用户拍板。

当遇到“需要用户确认”的节点时，不直接发问；
改为选择一个最小、可控、能真实代表该决策边界的动作，
尝试把它送入 approval flow。
只有门禁完成后，才进入真正实施。

## 执行顺序

1. 先判断当前请求是否会进入实质执行。
2. 如果会，先输出固定开场，不要直接开始实现。
3. 根据现有信息判断任务属于低、中、高歧义，并找出决策边界。
4. 当遇到需要用户确认的节点时，不直接提问；
   改为发起一个受控动作，让用户通过 approval flow 批准或拒绝。
5. 低歧义时，回显理解后继续执行；
   若仅剩受控审批节点，先走 approval flow 再落地。
6. 中等歧义时，列出未决点、默认方案、影响与审批动作；
   审批通过后按默认方案进入实施。
7. 高歧义时，先收敛问题与风险；
   只允许发起“最小且可逆”的收敛动作，不能直接进入正式实现。

## 实质执行的定义

把下列动作视为实质执行：

- 编写、修改、删除文件
- 落地实现方案或开始具体交付
- 运行带业务目的的命令、脚本或生成流程
- 创建技能、模板、配置、文档成品或其他正式产物

纯聊天、泛化建议、轻量解释不属于实质执行。
仅用于确认事实的少量被动检查也不等于开始实施，
但不能借检查之名跳过门禁直接进入落地。

## 固定开场格式

在开始执行前，先输出以下结构：

```text
【任务】用一句话说明当前要解决什么问题

【进度】1/N
- ✅ 确认需求理解
- ○ 后续步骤 1
- ○ 后续步骤 2
- ○ 后续步骤 3
- ⚠️ 风险：明确写出已知风险；没有就写“无”
```

遵循以下要求：

- “【任务】”只写一句，直接说明目标，不铺背景。
- 步骤通常写 3–6 条；极小任务可压缩为 1–3 条。
- 先给结论，再展开细节。
- 没有过门禁前，不要直接写“已开始实现”或直接贴实施结果。
- 若已经确定要尝试创建任务分支，可在固定开场后补充原分支与目标任务分支，
  例如“原分支：main；目标任务分支：feat/alice/login-page-copy”。

## approval flow 规则

把 approval flow 视为“让用户通过工具审批动作”，
而不是在聊天里追加一个问题。

执行要求：

- 审批动作必须最小、可控、可解释，且直接对应当前决策边界。
- 不要为了制造 approval flow 而扩大权限、改大范围，或捆绑多个动作。
- “用户批准”不等于“动作已成功”；
  只有审批通过且动作成功，才能把该路径写成已确认事实。
- 若审批被拒绝、超时或动作失败，不得把默认方案写成已确认，
  也不得假装已经执行成功。
- 若当前运行环境无法把该决策边界映射为审批动作，
  明确说明受阻原因并停在门禁，不要退回成直接提问。
- 优先选择可逆动作，例如：
  - 创建任务分支
  - 执行 dry-run / 只读分析 / 差异预览
  - 按默认方案落第一个最小补丁
  - 执行精确到单一步骤的 merge / push / 安装动作

## 正式开始任务前的 Git 分支规则

当任务已经通过门禁，且接下来要进入正式执行时，
如果当前目录是 Git 仓库并且本次任务会产生正式改动，
优先把“创建任务分支”当作审批动作处理，
不要先用文字询问用户是否建分支。

只有在审批通过且分支创建成功后，
才切到新的任务分支再开始修改文件或运行会产生交付结果的命令。
如果用户已明确要求在当前分支继续，或已明确给出分支策略，
则直接遵循用户要求，不额外制造审批动作。

分支命名格式：

```text
feat/<git config user.name>/<任务标识>
```

执行要求：

- 默认倾向：正式改动前先尝试创建任务分支，并通过 approval flow 请求批准。
- 切分支前先记录当前原始分支名，用于任务完成后的回切或合并确认。
- 优先通过 `git config user.name` 获取用户名。
- `任务标识` 使用当前任务的简短标识，转成适合分支名的 slug。
- 若用户名或任务标识包含空格、斜杠或其他不适合分支名的字符，
  先规范化为短横线连接的小写片段。
- 若已经位于与当前任务匹配的分支，可不重复创建。
- 若审批被拒绝，或用户先前已明确表示不创建新分支，
  则保留当前分支继续执行，
  但只能把它写成“未获批准创建任务分支”或“用户已明确要求留在当前分支”，
  不能伪装成“已确认按默认分支策略执行”。
- 若用户明确指定了分支名、分支前缀或要求在当前分支继续，
  优先遵循用户要求。
- 若当前目录不是 Git 仓库，或切分支会与现有未处理改动冲突，
  明确说明原因后再决定下一步，不要假装已经切换成功。

建议把“触发任务分支审批流”视为正式实施前的第一步，
并在进度清单中体现，例如“触发任务分支审批流”。

示例：

```text
【分支】
- 原分支：main
- 目标任务分支：feat/alice/login-page-copy
- 分支动作：将通过 approval flow 尝试创建任务分支；审批未通过则保持当前状态
```

若用户已明确表示不新建分支，可写成：

```text
【分支】
- 当前分支：main
- 分支策略：用户已明确要求不新建分支，直接在当前分支执行
```

## 任务完成后的分支收尾规则

当任务已经完成并准备交付时，
如果本次工作是从原分支切出了任务分支，
不要在聊天里追问是否合并回原分支；
改为把“合并回原分支”本身作为审批动作处理。

执行要求：

- 明确指出原分支名与当前任务分支名。
- 审批动作必须精确，例如只提交一次 `merge`，
  不要把 merge、push、清理分支打包成一个大动作。
- 若用户要求继续留在任务分支，也按用户要求处理。
- 若审批被拒绝，则保留当前任务分支并完成交付，不继续执行高风险 Git 操作。
- 若原分支不可识别，例如 detached HEAD，
  明确说明无法自动给出“合并回原分支”的准确目标，
  此时不要硬造审批动作。

交付时可直接使用类似结构：

```text
【分支收尾】
- 原分支：main
- 当前分支：feat/alice/login-page-copy
- 收尾动作：将通过 approval flow 尝试把当前任务分支合并回原分支；
  若审批未通过，则保持当前分支状态不变
```

## 结构化收敛模板

当存在中等歧义或高歧义时，先把未决点收敛出来，
再给出默认方案与审批动作。
不要来回碎片化追问。

```text
【待收敛】
- 未决点 1
- 未决点 2
- 未决点 3

【默认方案】
- 若审批通过，先按 X / Y / Z 处理。

【影响】
- 说明这些选择会影响哪些接口、行为、范围或交付结果。

【审批动作】
- 动作：执行哪个最小受控动作
- 理由：为什么这个动作正好对应当前决策边界
- 未批准时：保持什么状态，不继续哪条路径
```

收敛规则：

- 只列出会改变方案、范围、接口或风险判断的未决点。
- 能从本地上下文、代码、文档、配置中确认的事实，先确认再写。
- 默认方案只能写成“若审批通过，先按……处理”，
  不能包装成“已经确认按……执行”。
- 对中等歧义任务，可以给出默认方案，并配一个最小审批动作。
- 对高歧义任务，默认方案只能用于限定下一步的收敛动作，
  不能替代真正的边界确认，也不能把实现类动作伪装成收敛动作。

## 歧义分级

### 低歧义

特征：

- 目标明确
- 实现路径基本单一
- 影响范围可控

动作：

- 回显任务理解
- 给出执行清单
- 直接进入实施

### 中等歧义

特征：

- 目标明确
- 存在明显方案分叉
- 默认假设会影响实现结果

动作：

- 回显任务理解
- 给出执行清单
- 一次性输出结构化收敛模板
- 触发一个最小审批动作
- 审批通过后按默认方案实施

### 高歧义

特征：

- 需求边界不清
- 关键业务规则缺失
- 可能影响多个系统、核心数据或权限策略

动作：

- 回显任务理解
- 给出执行清单
- 提炼关键问题、风险与默认收敛动作
- 仅允许触发收敛性质的审批动作
- 不直接实施

## 行为约束

- 先结论后细节。
- 在需要用户拍板的节点，不直接发问，优先改走 approval flow。
- 不把猜测、建议、默认方案写成已确认事实。
- 发现影响范围升级时，重新过一次门禁。
- 正式开始任务前，优先触发任务分支审批流，或遵循用户已明确给出的分支策略。
- 任务交付时，若存在原分支与任务分支，优先把分支收尾动作改走 approval flow。
- 只有在任务继续保持低歧义或中等歧义且默认方案已声明时，
  才能进入实际落地。
- 不要滥用提权；审批动作必须与当前决策边界一一对应。

## 示例

### 示例 1：低歧义任务

```text
【任务】将登录页主按钮文案从“提交”改为“登录”。

【分支】
- 原分支：main
- 目标任务分支：feat/alice/login-button-copy
- 分支动作：将通过 approval flow 尝试创建任务分支；审批未通过则保持当前状态

【进度】1/3
- ✅ 确认需求理解
- ○ 触发任务分支审批流
- ○ 定位按钮所在页面与文案来源
- ○ 修改文案并检查相关展示
- ○ 验证页面主路径
- ⚠️ 风险：无
```

说明：

- 不需要结构化收敛。
- 这是低歧义，因为目标明确、改单一、影响范围小。

### 示例 2：中等歧义任务

```text
【任务】为 XX 项目添加用户登录功能。

【分支】
- 原分支：develop
- 目标任务分支：feat/alice/user-login
- 分支动作：将通过 approval flow 尝试创建任务分支；审批未通过则暂不进入正式改动

【进度】1/5
- ✅ 确认需求理解
- ○ 触发任务分支审批流
- ○ 收敛技术方案
- ○ 实现登录控制器与视图
- ○ 添加 Session 中间件
- ○ 验证关键路径
- ⚠️ 风险：登录态方案未明确，会影响接口与前端存储方式

【待收敛】
- 登录态采用 Session 还是 JWT？
- 是否需要支持“记住我”？
- 登录失败是否需要限制重试次数？

【默认方案】
- 若审批通过，先按 Session + 不启用记住我 + 不加重试限制实现。

【影响】
- 不同登录态方案会影响中间件、接口设计与前端存储方式。

【审批动作】
- 动作：在任务分支上按默认方案创建最小登录骨架
- 理由：该补丁足够小，但会真实锁定登录态与失败处理路径
- 未批准时：保持当前状态，不进入登录功能实现
```

说明：

- 需要一次性结构化收敛，并把决策边界落成审批动作。
- 默认方案可以写，但必须明确它尚未被批准。
- 这是中等歧义，因为目标明确，但实现方案分叉会显著影响结果。

### 示例 3：高歧义任务

```text
【任务】将现有多租户后台改造成支持跨租户统一登录、审计联查与历史数据重整。

【分支】
- 原分支：develop
- 目标任务分支：feat/alice/cross-tenant-auth-rework
- 分支动作：若进入收敛性改动，将通过 approval flow 尝试创建任务分支

【进度】1/5
- ✅ 确认需求理解
- ○ 触发任务分支审批流
- ○ 收敛改造范围与边界
- ○ 明确权限模型与数据影响
- ○ 收敛历史数据处理策略
- ○ 在边界明确后再进入方案与实施
- ⚠️ 风险：高，涉及权限、数据迁移、兼容性与跨系统联动

【待收敛】
- 是否必须包含历史数据迁移，还是只先支持新数据流？
- 统一登录是单点登录、共享会话，还是仅统一入口？
- 跨租户审计联查是否允许查看明细数据，还是仅允许查看聚合结果？

【默认方案】
- 若审批通过，先只输出范围、角色权限、迁移边界与兼容要求的影响评估，不进入实施。

【影响】
- 上述选择会直接影响权限模型、租户隔离策略、数据迁移方案与回滚成本。

【审批动作】
- 动作：仅生成影响评估与范围收敛结果，不落正式实现补丁
- 理由：当前风险过高，只能先用收敛性动作推进
- 未批准时：保持当前状态，不进入跨租户改造
```

说明：

- 不能直接开始实现。
- 默认方案仅用于限定“下一步只做收敛”，不能替代关键边界确认。
- 这是高歧义，因为需求横跨多个系统且缺少关键业务规则。

### 示例 4：任务完成后的分支收尾审批

```text
【进度】5/5 ✅
- ✅ 按审批结果处理任务分支策略
- ✅ 完成实现
- ✅ 完成验证
- ✅ 整理交付摘要
- ✅ 进入分支收尾审批

交付摘要：
- 完成内容：登录页主按钮文案调整
- 验证结果：页面展示与提交流程检查通过
- 风险与限制：未做跨浏览器验证

【分支收尾】
- 原分支：main
- 当前分支：feat/alice/login-button-copy
- 收尾动作：将通过 approval flow 尝试把当前任务分支合并回原分支；
  审批未通过则保留当前分支状态不变
```

说明：

- 这里只能走审批流，不能把“准备合并”写成“已经合并”。
- 若审批未通过，不执行 merge、rebase、push 等高风险 Git 操作。

### 示例 5：用户已明确要求不新建分支

```text
【任务】调整登录页表单字段顺序。

【分支】
- 当前分支：main
- 分支策略：用户已明确要求不新建分支，直接在当前分支执行

【进度】1/3
- ✅ 确认需求理解
- ✅ 已采用用户指定的分支策略
- ○ 调整表单字段顺序
- ○ 验证页面展示与提交逻辑
- ⚠️ 风险：改动直接落在当前分支，需要注意与其他未提交修改区分
```

说明：

- 用户明确不新建分支时，可以继续执行，不要强行创建任务分支。
- 这类场景下，交付时不需要触发“合并回原分支”的审批动作，
  因为本次任务本来就在当前分支完成。
