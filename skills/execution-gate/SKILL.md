# Execution Gate

## 概述

在开始任何实质执行前，先过“需求澄清门”，并先完成“规划确认”。
先输出一句任务理解和一组执行清单，再判断任务歧义等级，
识别哪些节点需要用户拍板，并把“本次准备如何推进”明确成可确认计划。

不要被技能名 `execution-gate` 误导：
本技能不仅用于“新增功能 / 新需求”，也用于“修改已有实现 / 调整现有逻辑 / 修复缺陷 / 继续未完成任务 / 改文档或配置”。
只要请求会进入实质执行，就应该触发本技能。

当遇到“需要用户确认”的节点时，不直接发问；
改为一次性给出互斥、可执行、影响明确的固定选项，
让用户直接在 A / B / C / D 中选择。
只有规划已确认且门禁完成后，才进入真正实施。

当规划已经确认后，后续默认按已定规划推进。
除非执行意图偏离已确认计划，或出现计划外的新边界 / 新风险，
否则尽量不要再次询问用户，也不要重复开启同一决策节点。

确认区只能输出 A / B / C / D 四项，不追加开放式追问。

如果后续执行意图偏离用户已明确表达的需求、范围、选项或决策，
必须重新向用户确认，不能擅自改写目标。

即使用户已经明确表示“不要继续询问”“你自己定”或等价意思，
也不得自行补齐尚未明确的未决点；
只能在用户已明确给出的需求、范围、选项和决策内继续执行，
或停在门禁等待边界进一步明确。

## 硬门禁

以下规则是硬约束，不是建议：

- 只要用户请求会进入实质执行，第一条回复必须先停在“规划确认”，不能在同条回复里直接实施。
- “规划确认”未完成前，禁止创建、修改、删除任何文件；禁止执行会产出正式结果的命令；禁止宣称“已完成实现”。
- 即使任务属于低歧义、实现极小、答案看起来显而易见，也必须先确认执行计划，不能跳过。
- 只有在用户明确认可当前计划后，才允许进入下一条实施消息。
- 若第一条回复已经开始写入文件、运行生成命令、贴出已落地结果或宣称已完成交付，应视为违反本技能。

把“先规划确认、后实施”理解为强制两阶段：

1. 第一阶段只允许输出任务理解、执行清单、风险、待确认方案、固定选项。
2. 第二阶段才允许真正实施，并且默认按已确认计划推进。

不要把“先简单做完再补一段计划说明”伪装成合规流程；
只要实施先于规划确认，就属于违规。

## 执行顺序

1. 先判断当前请求是否会进入实质执行。
2. 如果会，先输出固定开场，不要直接开始实现。
3. 先产出“任务理解 + 执行清单 + 风险”，把本次计划说清楚。
4. 根据现有信息判断任务属于低、中、高歧义，并找出决策边界。
5. 先完成规划确认；规划未确认前，不进入正式实施。
6. 当遇到需要用户确认的节点时，不直接提问；
   改为一次性输出固定选项，让用户直接选择 A / B / C / D。
7. 规划确认后，默认按已定规划推进；不要为计划内步骤重复询问用户。
8. 第一条回复若尚未获得规划确认，只能停在计划层，不能顺手执行任何一部分实施。
9. 若执行意图偏离用户已明确的需求、范围、选择或决策，
   必须先回到用户确认；不要擅自改写目标。
   若必须直接确认，则只输出 A / B / C / D 选项。
10. 若用户已明确表示该次会话内不要继续询问，
   也不得自行补齐尚未明确的未决点；
   只能继续执行已明确部分，或停在门禁。
11. 低歧义时，规划确认后继续执行。
12. 中等歧义时，列出未决点、待确认方案、影响与固定选项；
   用户选择后按已确认边界进入实施。
13. 高歧义时，先收敛问题与风险；
   只允许先收敛选择边界，不能直接进入正式实现。

## 实质执行的定义

把下列动作视为实质执行：

- 编写、修改、删除文件
- 落地实现方案或开始具体交付
- 运行带业务目的的命令、脚本或生成流程
- 创建技能、模板、配置、文档成品或其他正式产物

纯聊天、泛化建议、轻量解释不属于实质执行。
仅用于确认事实的少量被动检查也不等于开始实施，
但不能借检查之名跳过门禁直接进入落地。

## 固定开场格式

在开始执行前，先输出以下结构：

```text
【任务】用一句话说明当前要解决什么问题

【进度】1/N
- ✅ 确认需求理解
- ○ 确认执行计划
- ○ 后续步骤 1
- ○ 后续步骤 2
- ○ 后续步骤 3
- ⚠️ 风险：明确写出已知风险；没有就写“无”
```

遵循以下要求：

- “【任务】”只写一句，直接说明目标，不铺背景。
- 步骤通常写 3–6 条；极小任务可压缩为 1–3 条。
- 先给结论，再展开细节。
- 没有完成规划确认和门禁前，不要直接写“已开始实现”或直接贴实施结果。
- “确认执行计划”用于让用户先拍板本次推进路径；若用户已明确认可现有计划，
  后续默认沿该计划推进，不重复追问计划内步骤。
- 第一条回复必须以这段结构开场；不能把这段结构放在已经实施之后补写。
- 若已经确定要尝试创建任务分支，可在固定开场后补充原分支与目标任务分支，
  例如“原分支：main；目标任务分支：feat/alice/login-page-copy”。

## 固定选项确认格式

当需要用户选择时，统一使用以下格式：

```text
A. 方案 1
B. 方案 2
C. 方案 3
D. 用户自行填写
```

执行要求：

- 直接确认时，确认区只输出 A / B / C / D 四行。
- A / B / C 必须互斥、可执行、影响明确。
- D 固定保留给用户自行填写，不要省略。
- 不要输出“你想怎么做？”“还有别的要求吗？”之类开放式问法。
- 用户未选择前，不得按任一选项继续推进。

## Git 操作接入规则

当任务已经通过门禁，且接下来会产生正式改动时，
如果当前目录是 Git 仓库，必须显式使用 `$git-action`
处理任务分支、脏工作树、分支收尾以及高风险 Git 动作。

这样做的原因是：

- 避免在 `execution-gate` 内重复维护 Git 细则
- 把“需求澄清门”和“Git 动作门”拆成独立职责
- 当 Git 规则调整时，只需要维护 `$git-action`

执行要求：

- 在正式改动前，先判断是否需要调用 `$git-action`
- 只要涉及任务分支创建、切分支、脏工作树处理、merge、push、rebase、reset、stash 或分支收尾，就调用 `$git-action`
- 在进度清单中，可以保留“确认任务分支策略”这类步骤，但具体 Git 规则以 `$git-action` 为准
- 若用户已明确要求在当前分支继续，或已明确给出分支策略，也要按 `$git-action` 的记录方式呈现当前分支策略

建议在需要 Git 流程时，直接写出类似说明：

```text
【Git】
- 当前任务会产生正式改动，下一步将使用 $git-action 处理任务分支与后续 Git 选择边界
```

## 结构化收敛模板

当存在中等歧义或高歧义时，先把未决点收敛出来，
再给出待确认方案与固定选项，并把它作为“规划确认”的主体。
不要来回碎片化追问。

```text
【待收敛】
- 未决点 1
- 未决点 2
- 未决点 3

【选择】
A. 方案 1
B. 方案 2
C. 方案 3
D. 用户自行填写

【待确认方案】
- 当前推荐方案：X / Y / Z。

【影响】
- 说明这些选择会影响哪些接口、行为、范围或交付结果。
```

收敛规则：

- 只列出会改变方案、范围、接口或风险判断的未决点。
- 能从本地上下文、代码、文档、配置中确认的事实，先确认再写。
- 若必须直接向用户拍板，把候选方案压缩成 A / B / C / D；
  不要使用开放式问法。
- 待确认方案只能写成“当前推荐方案是……”，
  不能包装成“已经确认按……执行”。
- 一旦该方案已获确认，后续默认按它推进；
  不要对同一计划内子步骤反复向用户确认。
- 若执行意图偏离用户已明确的需求、范围、选择或决策，
  必须重新向用户确认。
- 若用户已明确表示该次会话内不要继续询问，
  也不得把未决点升级成其他替代性方案继续推进。
- 对中等歧义任务，可以给出待确认方案和固定选项。
- 对高歧义任务，待确认方案只能用于限定下一步的收敛动作，
  不能替代真正的边界确认，也不能把实现类动作伪装成收敛动作。

## 歧义分级

### 低歧义

特征：

- 目标明确
- 实现路径基本单一
- 影响范围可控

动作：

- 回显任务理解
- 给出执行清单
- 完成规划确认后直接进入实施

### 中等歧义

特征：

- 目标明确
- 存在明显方案分叉
- 不同方案会显著影响实现结果

动作：

- 回显任务理解
- 给出执行清单
- 一次性输出结构化收敛模板，完成规划确认
- 等待用户选择
- 用户选择后按已确认边界实施

### 高歧义

特征：

- 需求边界不清
- 关键业务规则缺失
- 可能影响多个系统、核心数据或权限策略

动作：

- 回显任务理解
- 给出执行清单
- 提炼关键问题、风险与收敛动作，先完成规划收敛
- 仅允许收敛选择边界
- 不直接实施

## 行为约束

- 先结论后细节。
- 在需要用户拍板的节点，不直接开放式发问，只输出固定选项。
- 不把猜测、建议、待确认方案写成已确认事实。
- 任务开始时先确认规划；规划确认后，默认按计划推进。
- 规划内步骤尽量不要再次询问用户；只有在偏离已定规划时才回到确认。
- 第一条回复禁止直接实施；低歧义任务也不能例外。
- 若执行意图偏离用户已明确的需求、范围、选择或决策，必须重新确认用户选择；
  若为直接确认，则只输出 A / B / C / D。
- 若用户已明确表示该次会话内不要继续询问，
  也不得自行补齐尚未明确的未决点或生成替代性方案。
- 发现影响范围升级时，重新过一次门禁。
- 正式开始任务前，如需进入 Git 流程，优先调用 `$git-action`。
- 任务交付时，若存在原分支与任务分支，分支收尾规则也通过 `$git-action` 承接。
- 只有在任务继续保持低歧义或中等歧义且关键边界已明确时，
  才能进入实际落地。
- 固定选项必须与当前决策边界一一对应。

## 示例

### 示例 1：低歧义任务

```text
【任务】将登录页主按钮文案从“提交”改为“登录”。

【Git】
- 当前任务会产生正式改动，下一步将使用 $git-action 处理任务分支与选择边界

【进度】1/5
- ✅ 确认需求理解
- ○ 确认执行计划
- ○ 确认任务分支策略
- ○ 定位按钮所在页面与文案来源
- ○ 修改文案并检查相关展示
- ○ 验证页面主路径
- ⚠️ 风险：无
```

说明：

- 不需要结构化收敛。
- 第一条回复仍然只能停在计划确认，不能因为任务小就直接改文案。
- 这是低歧义，因为目标明确、改单一、影响范围小。

### 示例 2：中等歧义任务

```text
【任务】为 XX 项目添加用户登录功能。

【Git】
- 当前任务会产生正式改动，下一步将使用 $git-action 处理任务分支与选择边界

【进度】1/6
- ✅ 确认需求理解
- ○ 确认执行计划
- ○ 确认任务分支策略
- ○ 收敛技术方案
- ○ 实现登录控制器与视图
- ○ 添加 Session 中间件
- ○ 验证关键路径
- ⚠️ 风险：登录态方案未明确，会影响接口与前端存储方式

【待收敛】
- 登录态采用 Session 还是 JWT？
- 是否需要支持“记住我”？
- 登录失败是否需要限制重试次数？

【选择】
A. Session + 不启用记住我 + 不加重试限制
B. Session + 启用记住我 + 限制重试次数
C. JWT + 启用刷新令牌 + 限制重试次数
D. 用户自行填写

【待确认方案】
- 当前推荐方案：Session + 不启用记住我 + 不加重试限制。

【影响】
- 不同登录态方案会影响中间件、接口设计与前端存储方式。
```

说明：

- 需要一次性结构化收敛，并把决策边界压缩成固定选项。
- 计划确认后，后续默认按已确认方案推进，不再反复询问登录态细节。
- 待确认方案可以写，但必须明确它尚未被确认。
- 这是中等歧义，因为目标明确，但实现方案分叉会显著影响结果。

### 示例 2.1：用户要求本次会话内不要继续询问

```text
【任务】为 XX 项目添加用户登录功能。

【进度】1/5
- ✅ 确认需求理解
- ✅ 记录用户要求本次会话内不要继续询问
- ○ 确认当前无法完成规划确认
- ○ 整理已明确范围与风险
- ○ 停在门禁，等待关键边界明确
- ○ 边界明确后再进入实现
- ⚠️ 风险：登录态关键规则未明确，当前不能直接进入实现

【未决点】
- 登录态采用 Session 还是 JWT？
- 是否需要支持“记住我”？

【说明】
- 即使用户明确要求“不要继续询问”，也不能自行补齐上述未决点。
- 当前只能保留已确认范围，停在门禁，不进入登录实现。
```

说明：

- 该例外只适用于当前会话。
- 不能补齐用户尚未明确的未决点，也不能覆盖用户已经拍板的事项。

### 示例 3：高歧义任务

```text
【任务】将现有多租户后台改造成支持跨租户统一登录、审计联查与历史数据重整。

【Git】
- 若后续收敛动作会产生正式改动，将使用 $git-action 处理任务分支与选择边界

【进度】1/6
- ✅ 确认需求理解
- ○ 确认执行计划
- ○ 确认任务分支策略
- ○ 收敛改造范围与边界
- ○ 明确权限模型与数据影响
- ○ 收敛历史数据处理策略
- ○ 在边界明确后再进入方案与实施
- ⚠️ 风险：高，涉及权限、数据迁移、兼容性与跨系统联动

【待收敛】
- 是否必须包含历史数据迁移，还是只先支持新数据流？
- 统一登录是单点登录、共享会话，还是仅统一入口？
- 跨租户审计联查是否允许查看明细数据，还是仅允许查看聚合结果？

【选择】
A. 只支持新数据流 + 仅统一入口 + 仅允许查看聚合结果
B. 包含历史数据迁移 + 单点登录 + 仅允许查看聚合结果
C. 包含历史数据迁移 + 共享会话 + 允许查看明细数据
D. 用户自行填写

【待确认方案】
- 当前推荐方案：只输出范围、角色权限、迁移边界与兼容要求的影响评估，不进入实施。

【影响】
- 上述选择会直接影响权限模型、租户隔离策略、数据迁移方案与回滚成本。
```

说明：

- 不能直接开始实现。
- 待确认方案仅用于限定“下一步只做收敛”，不能替代关键边界确认。
- 这是高歧义，因为需求横跨多个系统且缺少关键业务规则。

### 示例 3.1：违规示例（不要这样做）

```text
【进度】3/3 ✅
- ✅ 确认需求理解
- ✅ 创建目标目录与文件
- ✅ 校验文件内容并交付
```

说明：

- 这类输出属于违规，因为它在第一条回复里直接进入实施，没有先完成规划确认。
- 即使任务只是创建一个很小的文本文件，也不能跳过“确认执行计划”。
- 若出现这种行为，说明技能约束没有被遵守，需要继续强化门禁或重新测试。

### 示例 4：任务完成后的分支收尾选择

```text
【进度】5/5 ✅
- ✅ 按选择结果处理任务分支策略
- ✅ 完成实现
- ✅ 完成验证
- ✅ 整理交付摘要
- ✅ 进入分支收尾选择

交付摘要：
- 完成内容：登录页主按钮文案调整
- 验证结果：页面展示与提交流程检查通过
- 风险与限制：未做跨浏览器验证

【Git】
- 本次任务存在任务分支时，将使用 $git-action 处理分支收尾选择；用户未选择前保持当前分支状态不变
```

说明：

- 这里只能等待用户选择，不能把“准备合并”写成“已经合并”。
- 若用户未选择，不执行 merge、rebase、push 等高风险 Git 操作。

### 示例 5：用户已明确要求不新建分支

```text
【任务】调整登录页表单字段顺序。

【Git】
- 用户已明确要求不新建分支；仍通过 $git-action 的记录格式呈现当前分支策略

【进度】1/4
- ✅ 确认需求理解
- ✅ 已采用用户指定的分支策略
- ✅ 确认执行计划
- ○ 调整表单字段顺序
- ○ 验证页面展示与提交逻辑
- ⚠️ 风险：改动直接落在当前分支，需要注意与其他未提交修改区分
```

说明：

- 用户明确不新建分支时，可以继续执行，不要强行创建任务分支。
- 这类场景下，交付时不需要触发“合并回原分支”的选择步骤，
  因为本次任务本来就在当前分支完成。
