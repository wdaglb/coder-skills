# Large File Gate

## Overview

在读取文件内容前先做体积门禁，避免把大文件直接塞进上下文。
默认规则是：先判断大小，再决定是否读取；大于 100KB 的文件禁止直接读取，必须先征求用户确认。

## Workflow

### 1. 先检查大小，不要直接打开

在任何全文读取动作之前，先用低成本方式确认文件大小，例如：

- `ls -lh`
- `wc -c`
- `stat`
- 目录遍历、文件名、扩展名、修改时间

把“读取文件内容”的动作理解为包括但不限于：

- `cat`
- `sed` 大段输出
- `less`
- 直接全文打开
- 脚本一次性读完整文件
- 任何等价的整文件加载方式

### 2. 小于等于 100KB 时再按需读取

如果文件大小小于等于 100KB：

- 可以继续读取
- 仍然优先只读最小必要片段
- 优先使用精确搜索、局部片段、目标段落，而不是无差别全文展开

### 3. 大于 100KB 时必须停下并询问用户

如果文件大小大于 100KB：

- 不要直接读取文件内容
- 先向用户说明为什么需要读取这个文件
- 明确说明该文件超过 100KB
- 等用户明确同意后再继续

可直接复用下面这种确认方式：

> 这个文件大小超过 100KB。当前为了完成任务，需要读取它的相关内容来确认 `[原因]`。是否允许我继续读取？

### 4. 用户未确认前的替代策略

在用户尚未确认前，只能使用低成本手段缩小范围，例如：

- 用 `rg` 搜索关键字，不展开大段上下文
- 只看目录结构和文件名模式
- 比较多个候选文件的大小、时间、类型
- 优先读取更小的相关文件
- 先读取索引、清单、配置、摘要类文件

目标是先证明“确实有必要读这个大文件”，而不是先读了再说。

### 5. 获得确认后仍然最小化读取

即使用户已经同意读取大文件，也不要默认整文件展开。

优先顺序：

1. 先搜索关键字
2. 再读取命中的最小相关片段
3. 只有证据表明确实不够时，再逐步扩大范围

## Hard Rules

- 不要跳过大小检查
- 不要把“只看一下”当作例外
- 不要通过脚本、编码转换、分块拼接等方式绕过门禁
- 不要因为文件是日志、JSON、CSV、SQL、Markdown、生成物或导出物就默认可读
- 不要把“可能有用”当作读取大文件的充分理由

## Response Pattern

当你决定不读取大文件时，直接说明：

- 哪个文件超过了 100KB
- 为什么当前不直接读取
- 你准备先用哪些低成本方式缩小范围
- 如果确实需要全文或片段读取，会先征求用户确认

当你获得确认后，继续时要说明：

- 你会先读哪一部分
- 为什么先读这一部分
- 仍然会尽量避免整文件展开
