# Git Action

## 概述

在任务会产生正式改动时，用这套规则统一处理 Git 仓库状态、任务分支、脏工作树与高风险 Git 动作。
目标是把 Git 决策收敛成最小、可回退、可审批的动作，避免把未确认的分支策略或高风险操作伪装成默认流程。

## 工作流

### 1. 先识别是否真的需要 Git 动作

满足下列任一条件时，进入本技能：

- 当前目录是 Git 仓库，且接下来会产生正式改动
- 需要创建、复用、切换、合并、提交、推送、暂存或清理分支
- 需要判断脏工作树是否会影响后续修改
- 其他技能要复用一套统一的 Git 审批规则

如果只是只读分析、解释代码、总结 diff、查看日志，且不会改变仓库状态，只做必要的只读检查，不必强行套完整分支流程。

### 2. 在改动前检查仓库状态

在任何会改变仓库状态的 Git 动作前，先确认：

- 当前目录是否是 Git 仓库
- 当前所在分支名是否可识别
- 工作树是否存在未提交改动
- 用户是否已经明确给出分支策略

优先先用只读命令收集证据，再决定是否需要审批动作。

### 3. 任务开始前优先处理任务分支

当任务已经通过上游门禁且接下来会产生正式改动时：

- 用户已明确要求在当前分支继续，直接遵循
- 用户已明确指定分支名或分支前缀，优先使用
- 否则，优先把“创建任务分支”当作审批动作处理，而不是先在聊天里追问

分支命名默认格式：

```text
feat/<git config user.name>/<任务标识>
```

命名要求：

- 优先通过 `git config user.name` 获取用户名
- `任务标识` 使用当前任务的简短标识
- 若用户名或任务标识包含空格、斜杠或其他不适合分支名的字符，统一规范化为短横线连接的小写片段
- 若当前已经位于与任务相匹配的分支，可复用当前分支，不重复创建

### 4. 先处理脏工作树，再决定是否切分支

切换或创建分支前，必须先检查工作树是否存在未提交改动。若存在未提交改动：

- 不要直接执行 `git switch`、`git checkout`
- 不要直接创建并切入新分支
- 先明确告知用户当前存在未提交代码，再按固定选项要求用户选择处理策略

固定选项：

```text
A. 保留未提交代码，继续在当前分支完成本次任务
B. 先提交当前未提交代码，再切换到任务分支
C. 先暂存当前未提交代码，再切换到任务分支
D. 用户自行填写
```

执行要求：

- 在用户明确选择前，不得擅自假设要提交、暂存或继续切分支
- 只有当当前工作树状态已经与用户选择一致，且不会误带未提交改动进入目标分支时，才能继续切分支
- 如果用户选择继续在当前分支完成任务，后续交付时不要伪装成已经遵循默认任务分支策略

### 5. 用 approval flow 处理最小 Git 决策边界

把 approval flow 视为“让用户审批一个最小且直接对应边界的 Git 动作”，不要在聊天里追加开放式问题。

适合作为审批动作的场景包括：

- 创建任务分支
- 执行单一步骤的 `merge`
- 执行一次明确目标的 `push`
- 执行一次明确范围的安装或同步动作（若该动作与 Git 流程直接耦合）

执行要求：

- 一个审批动作只承载一个明确 Git 决策
- 不要把 `merge + push + 删除分支` 打包成单个动作
- 审批通过不等于动作已成功；动作失败时必须如实回报失败
- 审批被拒绝、超时或失败时，不得把待确认方案写成已确认事实

### 6. 任务完成后优先处理分支收尾

如果本次工作从原分支切出了任务分支，交付时：

- 明确指出原分支名与当前任务分支名
- 不要直接在聊天里追问“是否要合并回去”
- 改为把“合并回原分支”本身作为审批动作处理

收尾要求：

- 审批动作必须精确到单一步骤，例如一次 `merge`
- 若用户要求继续留在任务分支，按用户要求处理
- 若审批被拒绝，则保留当前任务分支并完成交付，不继续执行高风险 Git 操作
- 若原分支不可识别，例如 detached HEAD，明确说明无法自动确定合并目标，不要硬造审批动作

交付时可使用以下结构：

```text
【分支收尾】
- 原分支：main
- 当前分支：feat/alice/login-page-copy
- 收尾动作：将通过 approval flow 尝试把当前任务分支合并回原分支；
  若审批未通过，则保持当前分支状态不变
```

### 7. 高风险 Git 动作必须显式受控

以下操作不应被当作默认后续步骤自动执行：

- `push`
- `rebase`
- `reset`
- `force` 系列动作
- 删除分支或清理关键历史

处理原则：

- 先说明目标、影响与回退边界
- 再把动作拆成最小可审批步骤
- 若审批未通过，不继续执行该高风险动作

### 8. 提交消息要概括实际功能改动

当需要创建 commit 时，提交消息必须概括“这次提交实际完成了什么”，不要偷懒写成空泛描述。

执行要求：

- 提交少量文件时，直接概括本次功能、修复或重构点
- 提交大量文件时，先快速归纳这些改动共同完成的功能主题，再写成短消息
- 不要只写“提交所有代码”“提交全部变更”“更新代码”“同步修改”这类无信息量消息
- 不要机械罗列文件名；优先描述用户能理解的功能结果
- 消息要短，但必须能看出这次提交主要实现了什么

可接受方向示例：

- `fix: 修复 rpc 调试页抽屉长 JSON 无法滚动`
- `feat: 新增后台账号密码登录`
- `refactor: 统一权限校验入口并兼容旧角色`

## 响应模板

### 任务开始前的分支说明

```text
【分支】
- 原分支：main
- 目标任务分支：feat/alice/login-page-copy
- 分支动作：将通过 approval flow 尝试创建任务分支；审批未通过则保持当前状态
```

如果用户明确要求不新建分支：

```text
【分支】
- 当前分支：main
- 分支策略：用户已明确要求不新建分支，直接在当前分支执行
```

### 脏工作树确认模板

```text
A. 保留未提交代码，继续在当前分支完成本次任务
B. 先提交当前未提交代码，再切换到任务分支
C. 先暂存当前未提交代码，再切换到任务分支
D. 用户自行填写
```

### 任务完成后的分支收尾说明

```text
【分支收尾】
- 原分支：main
- 当前分支：feat/alice/login-page-copy
- 收尾动作：将通过 approval flow 尝试把当前任务分支合并回原分支；
  审批未通过则保留当前分支状态不变
```

### 提交前消息要求

```text
【提交消息】
- 先概括本次提交实际完成的功能、修复或重构点
- 消息保持简短，但不要写成“提交所有代码”“提交全部变更”“更新代码”
```

## 行为约束

- 先确认事实，再决定 Git 路径；不要根据习惯猜测仓库状态
- 不要在存在未提交改动时直接切分支
- 不要把用户未确认的分支策略包装成“默认已确认”
- 不要把多个 Git 决策捆绑成一个审批动作
- 不要把高风险 Git 动作藏进“顺手做掉”的实现步骤
- 不要使用“提交所有代码”“提交全部变更”“更新代码”“同步修改”这类空泛 commit 消息
- 若当前目录不是 Git 仓库，或 Git 状态与预期冲突，必须明确说明受阻原因
