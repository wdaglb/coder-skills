# Ambiguity Planner

## 概述

当任务已经被判断为中等歧义或高歧义时，使用本技能把规划过程拆成“当前一步 -> 用户选择 -> 下一步”的收敛链路。
每一步都必须说明当前推荐做法、固定选项，并在用户完成该步选择后，再判断是否需要继续下一步或已经可以进入实施；不要在用户选择前单独输出“影响”分节，避免让用户误以为这些影响已经被确认。
收敛深度由任务复杂度和用户当前这一步的选择共同决定，不预设固定只有 2 步或 3 步；只要仍存在关键未决边界，就继续往下收敛。
若当前已经知道“缺什么边界才能实施”，也不能只把缺口原样抛回给用户；必须先基于已收集上下文，把该缺口压缩成用户可直接选择的方案集合，再让用户选。
若需求仍不清楚、关键边界仍不足以稳定实施，就继续推进下一轮收敛，不因“已经问过一轮”就草率结束；但若问题已经简单到不会改变方案、接口、风险或验证策略，也不要为了走技能流程硬拆成多轮提问。

本技能只负责中高歧义任务的分步澄清与规划，不负责替代统一门禁。
若任务仍处于“是否允许项目外扩散”“是否允许高风险 Git 动作”等边界确认阶段，仍应遵循 `$execution-gate` 与 `$git-action` 的规则。
当分步规划已经足够明确、准备结束提问并回到实施前，应先询问用户是否要落实到方案文件；若用户确认，则自动检查 `docs/plans`，目录不存在时创建目录，再写入方案文件。

## 适用边界

在以下场景使用本技能：

- 已确认任务会进入实质执行，但当前方案存在关键分叉。
- 不同方案会显著影响接口、数据结构、权限模型、兼容性、测试范围或交付边界。
- 需要先问清第 1 个关键问题，用户答完后，下一步问题才有意义。
- 需要明确推荐做法与理由，而不是只列选项。
- 需要把“下一步先做什么”收窄到单个关键问题。

在以下场景不要使用本技能：

- 低歧义任务，且不存在真正会改变实现路径的关键未决点。
- 用户只是要解释、总结、翻译、复盘。
- 任务已经完成规划确认，只剩按既定计划实施。
- 当前唯一待确认点属于项目外扩散、高风险 Git、破坏性变更等门禁边界；这些边界应先由 `$execution-gate` 处理。
- 虽然有少量信息缺口，但基于现有上下文与默认约定已足以低风险推进，不需要专门开启分步收敛。

## 核心工作流

### 1. 先识别当前最关键的一步

- 不要试图一次性列完所有后续问题。
- 只挑出当前最影响后续路径、最应该先拍板的那个问题。
- 若当前问题没有被确认，后续问题会失去意义或需要重写，就说明它是优先步骤。
- 若当前问题已经小到不影响方案分叉，只影响局部实现细节或默认参数，就不要把它提升成单独步骤。

### 2. 输出“当前步骤”而不是整棵决策树

统一使用类似结构：

```text
【当前步骤】1/2
- 用一句话说明当前只确认哪个关键问题。

【选择】
A. 方案 1
B. 方案 2
C. 方案 3
D. 用户自行填写

【当前推荐】
- 当前推荐方案：X。
```

要求：

- 只输出当前步骤对应的 A / B / C / D。
- A / B / C 必须互斥、可执行、边界明确。
- A / B / C 必须是“用户可直接拍板的方案”，不能只是把开放问题换成三句不同表述。
- 不要追加开放式追问。
- `D. 用户自行填写` 固定保留。
- 步骤格式必须写成具体的 `当前步数/当前估计总步数`，例如 `1/2`、`2/3`，不要写 `1/N` 这类占位形式。
- 总步数必须基于当前已知信息做出归纳估计；如果用户后续选择引入新边界或收敛掉旧问题，可以在下一轮把总步数改写为新的具体数字，但每一轮都必须给出当下的具体估计。
- 若当前最关键问题是“范围”与“交付出口”这类双维度边界，应优先选择更能锁定实施路径的那个维度作为当前步骤；另一个维度留到下一步，而不是把两个开放问题一起甩给用户。
- 若已有上下文足以归纳常见方案，必须先给出 A / B / C，再允许 `D. 用户自行填写`；不能只写“请说明你要哪种边界”。
- 只有在证据不足到无法形成互斥选项时，才允许先补做被动上下文收集；此时要明确缺失证据是什么、准备去哪里确认，而不是直接要求用户补完整方案。

### 3. 先给推荐，再解释为什么

- 推荐方案必须明确写出，但不能包装成“已确认”。
- 推荐必须说明理由，例如：
  - 对当前系统改动更小
  - 更符合现有架构
  - 回滚成本更低
  - 测试面更可控
  - 可以把高风险决策延后
- 不要只报结论，不解释推荐依据。
- 若 A / B / C 的风险显著不同，推荐理由后应顺带点明“为什么不优先另外两项”，至少说明对实现范围、兼容性、回滚成本、测试面中的一个关键差异。

### 4. 影响后置到用户选择之后

- 在用户尚未选择前，不要单独输出 `【影响】` 分节，避免把待选项的差异包装成已经确认的结论。
- 用户完成当前步骤后，再在“已确认边界 / 下一步 / 实施前总结”中说明该选择实际带来的影响。
- 影响说明至少覆盖：
  - 接口与调用链
  - 数据结构 / 数据模型
  - 权限与安全边界
  - 兼容性与迁移成本
  - 测试范围
  - 后续是否还需要继续收敛
- 若影响暂不确定，明确写“当前尚未确认，需要在下一步或代码上下文中继续确认”。

### 5. 用户完成当前步骤后，再决定下一步

- 用户选完后，先判断是否还有新的关键未决点。
- 若用户选择 `D. 用户自行填写`，先把用户自填内容重述为“当前已确认边界”，再基于该边界决定下一步问题。
- 用户选择 `D. 用户自行填写` 后，不要回退到原始 A / B / C 模板重问同一个问题；只有当用户自填内容仍不足以形成有效边界时，才允许继续收窄同一问题。
- 只要下一步仍能显著降低关键不确定性、缩小方案范围或锁定实施边界，就允许继续推进到下一步。
- 只有在确实还存在新的关键分叉时，才进入下一步收敛。
- 若当前仍有关键未决点，且这些未决点会直接阻断实现、验证或交付，就继续问，不要为了尽快结束而假定用户已接受推荐方案。
- 若规划已经足够明确，就停止提问，回到实施。
- 在停止提问、回到实施前，先询问用户是否要落实到方案文件；若用户确认，则写入 `docs/plans`。
- 不要为了“流程完整”硬凑第 2 步、第 3 步。
- 若任务复杂度高、边界层级多、系统影响广，允许继续进入第 4 步、第 5 步甚至更多轮收敛；只要每一步仍然是在解决新的关键问题，就是合理的。
- 用户某一步的选择可能直接收敛掉后续问题，使剩余步骤减少；也可能打开新的关键边界，使剩余步骤增加。
- 因此不要把前一轮写出的 `1/2`、`2/3` 视为固定承诺；每次收到用户选择后，都应重新判断剩余还需要几步，并给出新的具体总步数。
- 若连续推进后，新增问题已不能明显降低关键不确定性，只是在追问实现细节、偏好细枝或可在实施中验证的小差异，应停止继续深挖。
- 若继续追问只会把问题推向实现级、配置级、命名级或局部参数级细节，而不会改变方案、范围、接口、风险或验证策略，也应停止推进，转回实施。

### 6. 中等歧义与高歧义的区别

#### 中等歧义

- 目标明确。
- 关键分叉可能 1-2 轮即可收敛，但仍以实际复杂度和用户选择后的边界变化为准。
- 用户确认后，多数情况下可以直接进入实施。

处理方式：

- 优先只问 1 个最关键问题。
- 用户答完后，判断是否还需要第 2 个问题。
- 若第 2 个问题已经只是实现细节偏好，而不会显著改变路径，可直接按推荐方案推进实施。

#### 高歧义

- 需求边界不清。
- 缺少关键业务规则。
- 可能影响多个系统、核心数据、权限模型或兼容策略。

处理方式：

- 当前步骤通常不是“选实现方案”，而是“先收窄边界”。
- 推荐方案通常应偏保守，例如“先做范围/权限/迁移边界确认，不直接进入实现”。
- 在边界没有收窄前，不要伪装成已经完成规划。
- 如果连续一轮轮确认后，需求仍无法形成可实施边界，就继续问到关键边界闭合，而不是把未确认部分默认为“实施时再看”。

## 约束

- 不把猜测、建议、当前推荐写成已确认事实。
- 不一次性并列抛出所有后续问题。
- 不把“还缺什么边界”直接原样反问给用户；要先把边界整理成可选择的方案。
- 不把实现动作伪装成“收敛动作”。
- 不因为用户说“你自己定”“别再问了”就自动补齐真正未决的关键边界。
- 不把影响分析前置成用户尚未确认的既成事实。
- 不在没有必要时过度拆问；简单问题一轮就够时，不继续追问。
- 不把简单事项、默认约定事项或可在实施中顺手验证的小问题包装成“中高歧义规划”。
- 不在用户已经确认要落实方案文件后，直接跳过 `docs/plans` 的计划沉淀步骤进入实施。
- 不为了“看起来更严谨”而无限推进；若问题已经不能显著降低关键不确定性，应停止继续扩展。
- 不把收敛推进到过深层级；不要钻入不会改变方案、范围、接口、风险或测试策略的实现细节。
- 不输出“你要 A 还是 B？另外再补充 C 和 D”这类混合式问题；一轮只锁定一个关键问题，并把它压成固定选项。
- 不在关键边界仍未闭合时提前停止提问，更不能把“当前推荐”当成默认已确认答案。

## 输出验收清单

用于机械判断当前输出是否合格；不要凭“看起来差不多”放行。

### 合格输出至少满足

- 有明确的 `【当前步骤】x/y`，且只锁定一个当前最关键的问题。
- 有完整的 `【选择】` 区，并包含互斥、可执行的 `A / B / C / D`。
- 有明确的 `【当前推荐】`，且说明推荐理由。
- 推荐理由至少落到改动范围、兼容性、回滚成本、测试面、实现复杂度中的一个真实差异。
- 用户未选择前，不把某个方案的影响写成既成事实。
- 用户完成当前步骤后，才进入下一步或补充该选择带来的影响。
- 当需求仍不清楚到无法稳定实施时，会继续进入下一步，而不是停在“推荐如此，你看着办”。
- 当问题已经简单到不再改变实施路径时，会停止追问，不把简单事复杂化。

### 出现以下任一情况，直接判为违规输出

- 没有 `A / B / C / D` 固定选项。
- 没有 `【当前推荐】` 或只有结论没有理由。
- 一轮同时追问两个及以上关键边界，例如同时追问“交付出口”和“事件范围”。
- 把开放问题直接原样抛给用户，例如“你要哪种边界”“请补一句边界”“请说明要哪些范围”。
- 把推荐方案、推测方案或待确认方案写成已确认事实。
- 用户未选择前，就提前展开某个方案的接口、数据模型、兼容性影响，并默认按该方案继续推进。
- 选项不是互斥方案，而只是同一个开放问题的不同说法。
- 在当前步骤之外追加新的开放式追问，导致用户需要一次回答多个关键问题。

### 快速自检问法

在发送给用户前，至少自问这四句：

1. 我这轮是不是只在确认一个关键问题？
2. 用户能不能只选 `A / B / C / D` 就完成这一步，而不用自己重新组织方案？
3. 我有没有明确给出推荐和理由，而不是只丢问题？
4. 我有没有把还没确认的影响提前写成既成事实？
5. 如果现在还不能稳定实施，我是不是会继续问下一步，而不是草率收口？
6. 如果现在已经能稳定实施，我是不是及时停问，没有把简单事做复杂？

## 示例

### 示例 1：中等歧义任务，第 1 步

```text
【当前步骤】1/2
- 先确认登录态采用 Session 还是 JWT；这会决定后续中间件、接口与前端存储方式。

【选择】
A. Session
B. JWT
C. 先只做登录页与表单校验，暂不落登录态
D. 用户自行填写

【当前推荐】
- 当前推荐方案：A. Session。
- 推荐理由：对当前常规 Web 登录改动更小，也更贴近多数现有服务端登录体系。
```

### 示例 2：中等歧义任务，第 2 步

```text
【当前步骤】2/2
- 基于已确认的 Session 方案，继续确认是否支持“记住我”。

【已确认边界】
- 已确认登录态采用 Session；这一选择会直接影响中间件、接口设计与前端存储方式。

【选择】
A. 不支持“记住我”
B. 支持“记住我”，并延长 Cookie 生命周期
C. 本次只保留接口预留，不实现“记住我”
D. 用户自行填写

【当前推荐】
- 当前推荐方案：A. 不支持“记住我”。
- 推荐理由：范围更可控，安全策略更简单，当前更适合先完成主路径。
```

### 示例 2.1：用户选择 D 后，基于自填边界进入下一步

```text
【已确认边界】
- 用户自填：使用 HttpOnly Cookie 承载 JWT，并支持 30 天“记住我”。
- 上一步选择已经把登录态边界锁定为 JWT + HttpOnly Cookie，并引入长期登录续签与失效策略问题。

【当前步骤】2/2
- 基于已确认的“JWT + HttpOnly Cookie + 30 天记住我”方案，继续确认长期登录的续签与失效策略。

【选择】
A. 使用短期 Access Token + 长期 Refresh Token；登出时服务端回收 Refresh Token（推荐）
B. 使用单个长生命周期 JWT，不做 Refresh Token
C. 本次先保留 30 天记住我配置项，但暂不实现续签与登出失效
D. 用户自行填写

【当前推荐】
- 当前推荐方案：A. 使用短期 Access Token + 长期 Refresh Token；登出时服务端回收 Refresh Token。
- 推荐理由：安全边界更清晰，登出失效与续签路径更可控。
```

### 示例 3：高歧义任务，第 1 步

```text
【当前步骤】1/3
- 先确认这次改造是否包含历史数据迁移；若这一步未定，后续统一登录与审计联查的实现边界都无法稳定。

【选择】
A. 本期只覆盖新数据流，不做历史数据迁移
B. 本期包含历史数据迁移，但先只输出迁移方案与回滚方案
C. 本期直接实现历史数据迁移与配套改造
D. 用户自行填写

【当前推荐】
- 当前推荐方案：A. 本期只覆盖新数据流，不做历史数据迁移。
- 推荐理由：先把高风险迁移动作排除掉，能更快收窄当前实现边界。
```

### 示例 3.1：高歧义任务不能只做开放式追问

错误示例：

```text
你要的“通知事件接入”是只在网关里完成摄像头事件订阅并结构化记录，
还是还要把事件通过现有 RPC 主动上报到上游？
另外请一并说明事件范围，是全部智能事件还是只接某几类。
```

问题：

- 同时抛出了“交付出口”和“事件范围”两个关键问题，用户需要自己重新组织方案。
- 没有给出互斥可选项，也没有当前推荐，导致所谓“收敛”实际退化成开放式补需求。
- 用户即使回复一句话，也容易只回答其中一半，下一轮仍然无法稳定进入实施。

正确示例：

```text
【当前步骤】1/2
- 先确认事件接入后的交付出口；这一步会直接决定后续要不要改现有 RPC 主动上报链路。

【选择】
A. 先只做摄像头事件订阅，并在网关日志中结构化输出
B. 订阅事件后，通过现有 RPC 链路主动上报到上游
C. 同时支持“日志结构化输出 + RPC 主动上报”
D. 用户自行填写

【当前推荐】
- 当前推荐方案：A. 先只做摄像头事件订阅，并在网关日志中结构化输出。
- 推荐理由：先把事件订阅链路打通，改动范围最小、验证最直接，也能避免在上游事件协议尚未锁定时做出不兼容的主动上报实现。
```

说明：

- 先锁定“交付出口”，下一步再问“事件范围”，这样实施路径更稳定。
- 推荐方案不仅给结论，还交代了为什么此时不优先 B / C：它们都要求更早锁定上游协议和兼容边界。
