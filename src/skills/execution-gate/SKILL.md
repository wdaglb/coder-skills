# Execution Gate

## 概述

在开始任何实质执行前，先过统一门禁，并先判断任务属于轻歧义还是重歧义。
本技能负责三件事：

- 判断任务是否会进入实质执行
- 判断任务属于轻、重歧义
- 决定是直接用简化规划推进，还是转交 `$ambiguity-planner`

## 快速决策表

先按下面顺序做决定：

1. 若任务不会进入实质执行，本技能不主导后续实施。
2. 若轻歧义 checklist 全部满足，直接进入实施。
3. 若未满足轻歧义 checklist，统一按重歧义处理，并转交 `$ambiguity-planner`。
4. 若过程中出现项目外扩散、高风险 Git、破坏性边界，继续留在本技能内做边界确认。
5. 若重歧义规划已确认且用户要求落实方案文件，先写入 `docs/plans`，再实施。

不要让 `execution-gate` 承担全部规划细节：

- **轻歧义任务**：由本技能直接给出任务理解、推荐做法与风险后进入实施
- **重歧义任务**：显式调用 `$ambiguity-planner` 做分步收敛
- **项目外扩散 / 高风险 Git / 破坏性边界**：继续由本技能把关

## 职责对照

为了避免后续维护时重新把两个技能的边界混在一起，按下面的分工理解：

### execution-gate

- 负责所有实质执行前的统一门禁
- 判断任务是否属于轻 / 重歧义
- 轻歧义任务直接执行
- 重歧义任务转交给 `$ambiguity-planner`
- 继续把关项目外扩散边界
- 继续把关高风险 Git / 破坏性边界
- 轻歧义任务可直接进入实施；重歧义任务先收敛再实施
- 重歧义任务在规划确认后，应先询问用户是否落实到方案文件；用户确认后，再写入 `docs/plans`

### ambiguity-planner

- 只处理重歧义任务的规划收敛
- 每次只推进当前最关键的一步
- 每一步都要给出当前推荐、基于当前信息估计出的具体总步数，以及 A / B / C / D 固定选项
- 用户选择前不要单独输出 `【影响】` 分节；影响放到用户确认后的已确认边界或下一步说明中
- 用户确认一步后，再决定下一步
- 支持 `D. 用户自行填写` 后继续沿新边界收敛
- 步数不是固定值，会按任务复杂度和用户选择动态增减
- 一旦边界足够明确，就停止提问并回到实施

一句话分工：

- `execution-gate`：先拦、先分流、先守边界
- `ambiguity-planner`：再逐步收敛、逐步规划、逐步推进

## 硬门禁

以下规则是硬约束，不是建议：

- 若任务属于轻歧义，可以在说明任务理解、推荐做法与风险后直接实施，不需要额外询问用户决策。
- 若任务属于重歧义，第一条回复必须先停在“规划确认/边界确认”，不能在同条回复里直接实施。
- 重歧义任务在确认完成前，禁止创建、修改、删除任何文件；禁止执行会产出正式结果的命令；禁止宣称“已完成实现”。
- 执行进行中若发现需要扩散到当前项目范围之外，必须先回到用户确认；未确认前，不得私自读取、修改、检索或排查项目外内容。
- 用户用自然语言说“你继续看”“不用再问我”“外面的也一起查”之类自由表述，不等于项目外扩散边界已经锁定；项目外边界仍必须压缩成 A / B / C / D 固定选项后再确认。
- 若重歧义任务在第一条回复已经开始写入文件、运行生成命令、贴出已落地结果或宣称已完成交付，应视为违反本技能。
- 若重歧义任务已经确认要落实方案文件，却未写入 `docs/plans` 就直接进入实施，应视为违反本技能。
- 若尚未实际改动文件、尚未生成结果，或尚未完成所声称的验证，不得使用“已修复”“修复已经落下”“完成实现”“最后做一次检查”等完成态表述；只能说“计划如此”“下一步将会”“待确认后执行”。
把“轻歧义可直做，重歧义先确认/收敛”理解为强制规则：

1. 轻歧义：先输出任务理解、推荐做法与风险，再直接实施。
2. 重歧义：第一阶段只允许输出任务理解、风险、当前计划或转交方案；确认后才允许真正实施。
3. 若重歧义任务已确认要落实方案文件，先写入 `docs/plans` 再实施。

把“真实状态一致”理解为强制约束：

1. 计划阶段只能陈述准备做什么，不能陈述已经做了什么。
2. 实施阶段只能汇报真实发生过的修改、命令和检查，不能提前写结果。
3. 验证阶段若仅做静态检查，就明确写“仅做静态检查，未做运行态/手工验证”；不要把静态检查写成完整验证。

## 执行顺序

1. 先判断当前请求是否会进入实质执行。
2. 如果会，先输出固定开场，说明任务理解、判断与风险。
3. 判断任务属于轻、重歧义。
4. 若为轻歧义，直接进入实施，不额外询问决策。
5. 若为重歧义，显式转交 `$ambiguity-planner`，不要在本技能内继续展开完整分步规划。
6. 若涉及项目外扩散、高风险 Git、破坏性边界，继续留在本技能内做边界确认。
7. 若为重歧义，且分步规划已经明确，先询问用户是否落实到方案文件。
8. 用户确认要落实方案文件后，自动检查 `docs/plans`；若目录不存在，则创建目录，再写入方案文件。
9. 方案文件落实后，默认按已定计划推进；不要为计划内步骤重复询问用户。
10. 若执行意图偏离用户已明确的需求、范围、选择或决策，必须重新向用户确认。

## 轻 / 重歧义判定

先按以下顺序判定，不要跳步：

1. 先查重歧义信号；只要命中任一重歧义信号，就直接判为重歧义。
2. 若未命中重歧义，再查轻歧义条件；只有轻歧义条件全部满足，才允许判为轻歧义。

默认升级规则：

- 判不准时，一律向上归类，不向下放行。
- 在“轻 vs 重”之间拿不准，判重。
- 只有证据足够支持更轻层级时，才允许降级。

### 快速 checklist

先按下面 checklist 机械判断，不要凭感觉自由发挥。

#### 重歧义 checklist

命中任一项，就判重歧义：

- 关键业务规则、权限规则、状态机规则或数据口径缺失
- 需要同时协调多个系统、多个仓库、多个服务或多个角色边界
- 会影响核心数据、权限模型、登录态、结算、审计、安全边界等高风险区域
- 存在明显项目外扩散、高风险 Git、线上数据、共享配置、公共组件等升级边界
- 存在方案分叉、边界不稳、主观取舍，或高层级风险开始主导实现路径

#### 轻歧义 checklist

必须全部满足，才判轻歧义：

- 目标明确
- 实现路径单一且自然收敛
- 影响范围可控
- 不存在关键未决点
- 不存在轻微或明显方案分叉
- 不需要主观取舍就能稳定落地
- 不涉及项目外扩散或高风险边界升级

若 checklist 与主观感觉冲突，以 checklist 为准。

### 判定落地规则

- 轻歧义 = 轻歧义 checklist 全部满足。
- 重歧义 = 其余全部情况。
- 一旦执行中发现新信号导致 checklist 变化，立即重新判级；从轻升级到重时，立刻重新过门禁。
- 不允许使用“先按轻歧义做做看”这类向下保守判法。

### 轻歧义

输出要求：

- 回显任务理解
- 说明推荐做法与主要影响
- 给出风险提示
- 直接进入实施

### 重歧义

常见触发：

- 规则缺失、边界不清、影响面不稳
- 存在方案分叉、主观取舍或需要先做一步收敛
- 涉及跨系统、核心数据、权限模型、登录态、结算、审计、安全边界等高风险区域
- 涉及项目外扩散、高风险 Git 或其他升级边界

输出要求：

- 回显任务理解
- 明确说明应转交 `$ambiguity-planner`
- 若尚有项目外扩散或高风险边界未锁定，先在本技能内完成边界确认
- 规划明确后，先询问用户是否落实到方案文件；若用户确认，则写入 `docs/plans`

## 固定开场格式

按判级选择以下固定开场之一：

### 轻歧义开场

```text
【实施中】用一句话说明当前要解决什么问题

【判断】轻歧义，可直接执行
【风险】明确写出已知风险；没有就写“无”
【做法】一句话说明当前推荐做法
```

### 重歧义开场

```text
【规划确认中】用一句话说明当前要解决什么问题

【判断】重歧义，先做收敛
【风险】明确写出已知风险；没有就写“无”
【规划】一句话说明当前最值得先确认的方向，并说明下一步将使用 $ambiguity-planner
```

遵循以下要求：

- 首行使用单标题格式：`【当前阶段】一句话任务目标`。
- 标题中的“当前阶段”只能从以下固定集合中选择：`规划确认中`、`实施中`、`验证中`、`已完成`。
- 不要自行扩写阶段标题，不要写成 `实施中：待确认项目外扩散边界`、`规划确认中-方案 A`、`验证中（静态）` 这类变体。
- 需要补充边界、条件、说明时，写到 `【判断】`、`【风险】`、`【做法】` 或 `【规划】` 中，不要塞进标题。
- 标题里的任务目标只写一句，直接说明目标，不铺背景。
- 默认使用“单标题阶段 + 判断/风险/做法/规划”的表达方式，不写步骤清单，不写数字进度。
- 先给结论，再展开细节。
- 轻歧义任务使用“轻歧义开场”；重歧义任务使用“重歧义开场”。
- 第一条回复必须以这段结构开场；不能把这段结构放在已经实施之后补写。

## 固定选项确认格式

当需要用户选择时，统一使用以下格式：

```text
A. 方案 1
B. 方案 2
C. 方案 3
D. 用户自行填写
```

执行要求：

- 直接确认时，确认区只输出 A / B / C / D 四行。
- A / B / C 必须互斥、可执行、影响明确。
- D 固定保留给用户自行填写，不要省略。
- 不要输出开放式追问。
- 用户未选择前，不得按任一选项继续推进。

## 重歧义任务转交规则

当任务属于重歧义时，显式交给 `$ambiguity-planner`。

转交时默认只保留三点：

- 当前任务为什么不是轻歧义
- 当前最值得先确认的方向是什么
- 下一步将由 `$ambiguity-planner` 继续逐步推进

优先使用以下短模板，不要随意展开成长段说明：

```text
【规划】
- 当前任务已超出轻歧义直接执行范围，先按重歧义处理。
- 下一步将使用 $ambiguity-planner 继续做分步收敛，先确认当前最关键的一步。
```

## Git 操作接入规则

当任务已经通过门禁，且接下来会产生正式改动时，
如果当前目录是 Git 仓库，必须显式使用 `$git-action`
处理任务分支、脏工作树、分支收尾以及高风险 Git 动作。

执行要求：

- 在正式改动前，先判断是否需要调用 `$git-action`
- 只要涉及任务分支创建、切分支、脏工作树处理、merge、push、rebase、reset、stash 或分支收尾，就调用 `$git-action`
- 回复中可以简述任务分支策略，但具体 Git 规则以 `$git-action` 为准

## 项目外扩散边界

如果执行进行中遇到问题，且下一步需要把排查、读取、修改或验证动作
扩散到当前项目之外的目录、仓库、服务、数据、环境或其他资源，
必须先停下并向用户确认。

确认要求：

- 不把“继续看”“别再问我”视为已授权
- 只用 A / B / C / D 固定选项确认
- 把“项目外只读排查”“允许修改外部仓库 / 配置”“允许直接操作线上数据”等不同风险层级分开

## 行为约束

- 先结论后细节。
- 不把猜测、建议、待确认方案写成已确认事实。
- 不要发明新的阶段标题；如果固定集合不够表达语义，就在进度项或风险项补充。
- 发现影响范围升级时，重新过一次门禁。
- 若用户已明确表示该次会话内不要继续询问，也不得自行补齐尚未明确的未决点。
- 不得把“尚未发生的修改”“尚未执行的验证”“预计能得到的结果”写成既成事实。
