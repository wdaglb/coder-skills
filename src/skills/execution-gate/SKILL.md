# Execution Gate

## 概述

在开始任何实质执行前，先过统一门禁，并先完成规划确认。
本技能负责三件事：

- 判断任务是否会进入实质执行
- 判断任务属于低、中、高歧义
- 决定是直接用简化规划推进，还是转交 `$ambiguity-planner`

不要让 `execution-gate` 承担全部规划细节：

- **低歧义任务**：由本技能直接做简化规划确认
- **中/高歧义任务**：显式调用 `$ambiguity-planner` 做分步收敛
- **项目外扩散 / 高风险 Git / 破坏性边界**：继续由本技能把关

## 职责对照

为了避免后续维护时重新把两个技能的边界混在一起，按下面的分工理解：

### execution-gate

- 负责所有实质执行前的统一门禁
- 判断任务是否属于低 / 中 / 高歧义
- 低歧义任务直接做简化规划确认
- 中 / 高歧义任务转交给 `$ambiguity-planner`
- 继续把关项目外扩散边界
- 继续把关高风险 Git / 破坏性边界
- 规划确认后才允许进入实施
- 中 / 高歧义任务在规划确认后，应先询问用户是否落实到方案文件；用户确认后，再写入 `docs/plans`

### ambiguity-planner

- 只处理中 / 高歧义任务的规划收敛
- 每次只推进当前最关键的一步
- 每一步都要给出当前推荐、影响分析与 A / B / C / D 固定选项
- 用户确认一步后，再决定下一步
- 支持 `D. 用户自行填写` 后继续沿新边界收敛
- 步数不是固定值，会按任务复杂度和用户选择动态增减
- 一旦边界足够明确，就停止提问并回到实施

一句话分工：

- `execution-gate`：先拦、先分流、先守边界
- `ambiguity-planner`：再逐步收敛、逐步规划、逐步推进

## 硬门禁

以下规则是硬约束，不是建议：

- 只要用户请求会进入实质执行，第一条回复必须先停在“规划确认”，不能在同条回复里直接实施。
- “规划确认”未完成前，禁止创建、修改、删除任何文件；禁止执行会产出正式结果的命令；禁止宣称“已完成实现”。
- 即使任务属于低歧义、实现极小、答案看起来显而易见，也必须先确认执行计划，不能跳过；但这类任务可以使用简化规划。
- 只有在用户明确认可当前计划后，才允许进入下一条实施消息。
- 执行进行中若发现需要扩散到当前项目范围之外，必须先回到用户确认；未确认前，不得私自读取、修改、检索或排查项目外内容。
- 用户用自然语言说“你继续看”“不用再问我”“外面的也一起查”之类自由表述，不等于项目外扩散边界已经锁定；项目外边界仍必须压缩成 A / B / C / D 固定选项后再确认。
- 若第一条回复已经开始写入文件、运行生成命令、贴出已落地结果或宣称已完成交付，应视为违反本技能。
- 若中 / 高歧义任务已经确认要落实方案文件，却未写入 `docs/plans` 就直接进入实施，应视为违反本技能。

把“先规划确认、后实施”理解为强制两阶段：

1. 第一阶段只允许输出任务理解、执行清单、风险、当前计划或转交方案。
2. 第二阶段才允许真正实施，并且默认按已确认计划推进；若中 / 高歧义任务已确认要落实方案文件，先写入 `docs/plans`。

## 执行顺序

1. 先判断当前请求是否会进入实质执行。
2. 如果会，先输出固定开场，不要直接开始实现。
3. 先产出“任务理解 + 执行清单 + 风险”，把本次计划说清楚。
4. 判断任务属于低、中、高歧义。
5. 若为低歧义，使用简化规划并等待确认。
6. 若为中等或高歧义，显式转交 `$ambiguity-planner`，不要在本技能内继续展开完整分步规划。
7. 若涉及项目外扩散、高风险 Git、破坏性边界，继续留在本技能内做边界确认。
8. 若为低歧义，规划确认后可直接按已定计划推进；低歧义简单任务不强制落实方案文件。
9. 若为中等或高歧义，且分步规划已经明确，先询问用户是否落实到方案文件。
10. 用户确认要落实方案文件后，自动检查 `docs/plans`；若目录不存在，则创建目录，再写入方案文件。
11. 方案文件落实后，默认按已定计划推进；不要为计划内步骤重复询问用户。
12. 若执行意图偏离用户已明确的需求、范围、选择或决策，必须重新向用户确认。

## 低歧义 vs 中高歧义

### 低歧义

特征：

- 目标明确
- 实现路径基本单一
- 影响范围可控
- 不存在真正会改变实现路径的关键未决点

动作：

- 回显任务理解
- 给出执行清单
- 说明推荐做法与主要影响
- 使用简化规划确认
- 规划确认后直接进入实施

### 中等歧义

特征：

- 目标明确
- 存在明显方案分叉
- 不同方案会显著影响实现结果

动作：

- 回显任务理解
- 给出执行清单
- 明确说明应转交 `$ambiguity-planner`
- 不要在本技能里继续把后续步骤全部展开
- 规划明确后，先询问用户是否落实到方案文件；若用户确认，则写入 `docs/plans`

### 高歧义

特征：

- 需求边界不清
- 关键业务规则缺失
- 可能影响多个系统、核心数据或权限策略

动作：

- 回显任务理解
- 给出执行清单
- 明确说明应转交 `$ambiguity-planner`
- 若尚有项目外扩散或高风险边界未锁定，先在本技能内完成边界确认
- 规划明确后，先询问用户是否落实到方案文件；若用户确认，则写入 `docs/plans`

## 固定开场格式

在开始执行前，先输出以下结构：

```text
【任务】用一句话说明当前要解决什么问题

【进度】1/N
- ✅ 确认需求理解
- ○ 确认执行计划
- ○ 后续步骤 1
- ○ 后续步骤 2
- ○ 后续步骤 3
- ⚠️ 风险：明确写出已知风险；没有就写“无”
```

遵循以下要求：

- “【任务】”只写一句，直接说明目标，不铺背景。
- 步骤通常写 3-6 条；极小任务可压缩为 1-3 条。
- 先给结论，再展开细节。
- 没有完成规划确认和门禁前，不要直接写“已开始实现”或直接贴实施结果。
- 第一条回复必须以这段结构开场；不能把这段结构放在已经实施之后补写。

## 固定选项确认格式

当需要用户选择时，统一使用以下格式：

```text
A. 方案 1
B. 方案 2
C. 方案 3
D. 用户自行填写
```

执行要求：

- 直接确认时，确认区只输出 A / B / C / D 四行。
- A / B / C 必须互斥、可执行、影响明确。
- D 固定保留给用户自行填写，不要省略。
- 不要输出开放式追问。
- 用户未选择前，不得按任一选项继续推进。

## 中高歧义任务转交规则

当任务属于中等或高歧义时，显式交给 `$ambiguity-planner`。

转交时要说明：

- 为什么当前任务不是低歧义
- 为什么需要分步收敛
- 当前最值得先确认的方向是什么
- 转交后由 `$ambiguity-planner` 继续逐步推进

可直接使用类似说明：

```text
【规划】
- 当前任务存在明显方案分叉/边界未定，已超出低歧义简化规划范围。
- 下一步将使用 $ambiguity-planner 继续处理中/高歧义任务的分步收敛：每次只确认当前最关键的一步，说明推荐做法、影响与 A / B / C / D 固定选项，用户确认后再进入下一步。
```

## Git 操作接入规则

当任务已经通过门禁，且接下来会产生正式改动时，
如果当前目录是 Git 仓库，必须显式使用 `$git-action`
处理任务分支、脏工作树、分支收尾以及高风险 Git 动作。

执行要求：

- 在正式改动前，先判断是否需要调用 `$git-action`
- 只要涉及任务分支创建、切分支、脏工作树处理、merge、push、rebase、reset、stash 或分支收尾，就调用 `$git-action`
- 在进度清单中，可以保留“确认任务分支策略”这类步骤，但具体 Git 规则以 `$git-action` 为准

## 项目外扩散边界

如果执行进行中遇到问题，且下一步需要把排查、读取、修改或验证动作
扩散到当前项目之外的目录、仓库、服务、数据、环境或其他资源，
必须先停下并向用户确认。

确认要求：

- 不把“继续看”“别再问我”视为已授权
- 只用 A / B / C / D 固定选项确认
- 把“项目外只读排查”“允许修改外部仓库 / 配置”“允许直接操作线上数据”等不同风险层级分开

## 行为约束

- 先结论后细节。
- 不把猜测、建议、待确认方案写成已确认事实。
- 第一条回复禁止直接实施；低歧义任务也不能例外。
- 发现影响范围升级时，重新过一次门禁。
- 任务开始时先确认规划；规划确认后，默认按计划推进。
- 中 / 高歧义任务在规划确认后，应先询问用户是否落实到方案文件；若用户确认，则自动检查并创建 `docs/plans` 后再写入方案文件。
- 若用户已明确表示该次会话内不要继续询问，也不得自行补齐尚未明确的未决点。

## 示例

### 示例 1：低歧义任务

```text
【任务】将登录页主按钮文案从“提交”改为“登录”。

【Git】
- 当前任务会产生正式改动，下一步将使用 $git-action 处理任务分支与选择边界

【进度】1/5
- ✅ 确认需求理解
- ○ 确认执行计划
- ○ 确认任务分支策略
- ○ 定位按钮所在页面与文案来源
- ○ 修改文案并检查相关展示
- ○ 验证页面主路径
- ⚠️ 风险：无

【规划】
- 当前推荐：直接在现有页面中做最小改动，不扩展到其他交互逻辑。
- 影响：仅影响页面展示文案，测试范围集中在展示与提交主路径。
```

### 示例 2：中等歧义任务转交

```text
【任务】为 XX 项目添加用户登录功能。

【进度】1/4
- ✅ 确认需求理解
- ○ 确认执行计划
- ○ 转交中等歧义规划
- ○ 在规划明确后进入实现
- ⚠️ 风险：登录态方案未明确，会影响接口、中间件与前端存储方式

【规划】
- 当前任务存在明显方案分叉，已超出低歧义简化规划范围。
- 下一步将使用 $ambiguity-planner 继续处理中等歧义任务的分步收敛。
- 分步规划确认完成后，将先询问用户是否落实到方案文件；若用户确认，则自动检查并创建 `docs/plans` 后写入方案文件，再进入实施。
```

### 示例 3：项目外扩散边界

```text
【任务】修复 XX 项目的登录超时问题。

【进度】2/6
- ✅ 确认需求理解
- ✅ 确认执行计划
- ✅ 已开始按确认方案定位项目内登录态链路
- ○ 确认项目外扩散边界
- ○ 在边界锁定后继续排查与修复
- ○ 验证正常路径与超时边界
- ⚠️ 风险：若继续扩散到公共认证仓库、共享网关或线上会话数据，影响范围会升级

A. 允许扩散到项目外做只读排查；外部仓库、共享配置、线上数据均不直接修改（推荐）
B. 允许扩散到项目外排查，并允许修改外部仓库或共享配置；线上数据仍不直接修改
C. 只允许继续留在当前项目内排查，不扩散到项目外
D. 用户自行填写
```
